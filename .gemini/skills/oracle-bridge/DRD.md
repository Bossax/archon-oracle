# Oracle Bridge: Stability & Sync Engine

**Signal ID**: SIG-20260415-0002
**Priority**: High
**Target Environment**: Windows (PowerShell)

## 1. What is the Oracle Bridge?
The Oracle Bridge is a maintenance tool and **Scaffolding Engine** that keeps your AI agent's environment stable and synchronized. It carries fleet-wide standards into a local workspace and executes the specific logic required to build the instruction and context files for different agents (Gemini, Roo Code, etc.).

**In short: It uses neutral "Master Mandates" to build agent-specific instruction systems.**

---

## 2. The "Ship-to-Sync" Model
To maintain Oracle sovereignty, the Bridge relies on a packaged deployment model:

1.  **Neutral Master Registry**: Shared rules are stored **inside the skill's own directory** (`assets/mandates/`) as universal Markdown files. They are NOT grouped by target agent.
2.  **The Shipping Act**: When Archon updates fleet standards, it modifies these universal templates and re-ships the `oracle-bridge` skill.
3.  **Local Scaffolding**: When an agent runs the Bridge, the skill executes its internal logic to transform these universal templates into the correct formats for the local agent.

---

## 3. The "Double-Layer" Strategy
*   **Layer 1 (The Source)**: Neutral, universal mandate templates bundled inside the `oracle-bridge` skill folder.
*   **Layer 2 (The Implementation)**: Agent-specific files generated by the Bridge logic in `.gemini/`, `.roo/`, and the project root.

---

## 4. Agent-Specific Scaffolding Logic
The skill contains "Drivers" that decide how to implement the universal mandates for each supported agent:

### For Gemini CLI (Hierarchical Driver)
*   **Logic**: 
    1. Create `.gemini/mandates/` if missing.
    2. Write universal mandates as modular files in that folder.
    3. Update or create `GEMINI.md` (The Router) with `@import` pointers to those files.
*   **Goal**: Minimize the root `GEMINI.md` while ensuring all mandates are natively discoverable.

### For Roo Code (Directory Driver)
*   **Logic**:
    1. Ensure `.roo/rules/` exists.
    2. Map universal mandates to numbered filenames (e.g., `01-system.md`, `02-identity.md`) to control loading order.
    3. Write the files directly into the rules directory.
*   **Goal**: Leverage Roo's native alphabetical rule aggregation.

---

## 5. What files does it manage?

| Component | Target File/Folder | What the Bridge does |
|-----------|--------------------|----------------------|
| **The Brain** | `ψ/` (all subfolders) | Scaffolds the 7 standard folders. |
| **Docker** | `docker-compose.yml` | Syncs container settings with bundled templates. |
| **Tools** | `settings.json` & `mcp.json` | Merges fleet tools into local configurations. |
| **Fleet Mandates** | `.gemini/` & `.roo/rules/` | Executes agent-specific logic to build instruction sets from neutral sources. |
| **Root Instruction** | `GEMINI.md` & `CLAUDE.md` | Injects the required "Pointers" or "Summaries" to ground the agent. |

---

## 6. How it Works (The Bridge Cycle)
1. **Identify**: Detects the agent identity and environment.
2. **Audit Brain**: Ensures the local `ψ/` pillars are intact.
3. **Execute Scaffolding**: 
    *   Reads neutral mandates from internal `assets/mandates/`.
    *   Applies **Gemini Driver** logic to build `.gemini/` and `GEMINI.md`.
    *   Applies **Roo Driver** logic to build `.roo/rules/`.
4. **Sync Tools**: Updates MCP and settings files.
5. **Report**: Logs all structural changes.

---

## 7. Core Mandates
* **Safety First**: Never overwrite a file without checking for custom user data.
* **Neutral Source**: Fleet mandates must be written as universal Markdown, free of agent-specific syntax.
* **Sovereignty**: No cross-oracle access. All information is carried within the skill package.
